# 设计方案: om:newfeature - 轻量级小需求指令

日期: 2026-04-18

## 核心目标

- 提供轻量级的小需求开发流程，无需完整任务文件管理
- 拆分为 2-5 个小任务块，内存状态管理
- 保留质量门禁验证（按质量等级执行）
- 支持分步 Git 提交
- AI 智能路由判断

## 背景与问题

**现有流程的问题：**
- `/om:start` 有 10 个步骤，包含完整的任务文件管理，对小需求过于重量级
- `/om:brainstorm` 用于复杂任务的设计探索，对小需求不必要
- `/om:auto` 绕过审批但流程结构相同，仍然产生文件

**用户场景：**
- 快速添加一个按钮、字段、简单功能
- 任务描述简短（≤100 字）
- 单一功能点，无需架构设计
- 希望快速迭代，不产生任务追踪文件

## 架构设计

### 指令路由

修改 `/om` 主入口 skill 的路由逻辑，AI 评估任务复杂度后自动路由：

```
用户输入 → /om skill → AI 评估复杂度 → 路由决策

小需求 (→ /om:newfeature)
• 任务描述 ≤ 100 字
• 单一功能点（无多个独立子系统）
• 不涉及架构设计/技术选型
• 关键词：小需求、小功能、小改动、minor、quick、快速、添加按钮

复杂任务 (→ /om:brainstorm)
• 多个独立子系统
• 需要架构设计/技术选型
• 从零搭建、重构、迁移

标准任务 (→ /om:start)
• 中等复杂度（介于小需求和复杂任务之间）
• 需要完整任务追踪
• 关键词：实现、修复、重构（非小改动）
```

### 文件结构

```
skills/
├── newfeature.md       # 新增 skill 文件
├── om.md               # 修改：增加路由逻辑和 newfeature 判断
└── openmatrix.md       # 修改：同步更新路由描述
```

## 流程设计

### /om:newfeature 流程

```
Step 1: 解析任务输入
        • 读取用户描述或任务文件
        • 分析任务边界（单一功能点确认）

Step 2: AI 拆分为 2-5 个小任务块
        • 拆分粒度：每个小块 ≤ 30 分钟工作量
        • 拆分原则：按逻辑边界（数据 → API → UI）
        • 输出格式：任务名称 + 简要描述 + 预估文件列表

Step 3: 用 TodoWrite 管理任务状态
        • 任务状态：pending → in_progress → completed
        • 不产生 `.openmatrix/tasks/` 文件
        • 状态仅在当前会话有效

Step 4: 问答确认
        • 质量等级：strict/balanced/fast
        • E2E 测试：功能测试/视觉验证/不需要
        • 展示拆分计划让用户确认

Step 5: 逐个执行小任务
        • 调用 Agent 工具（subagent_type: "general-purpose"）
        • 传递任务描述 + 预估文件 + 质量等级要求
        • 每个任务完成后执行验证 + Git 提交

Step 6: 验证（按质量等级）
        strict:   测试 + lint + 覆盖率 > 80%
        balanced: 测试 + lint + 覆盖率 > 60%
        fast:     无验证

        E2E（如果用户选择）：
        功能测试：运行 playwright/cypress 测试
        视觉验证：启动浏览器截图对比

Step 7: 分步 Git 提交
        • 每个小任务完成后立即提交
        • 提交格式：feat(newfeature): 添加搜索功能 - 数据层

Step 8: 全部任务完成后
        • 检查所有 TodoWrite 任务状态为 completed
        • 运行最终验证
        • 生成执行摘要

Step 9: 输出摘要
        • 列出所有创建/修改的文件
        • 关键决策总结
        • 测试覆盖率统计

Step 10: 清理
        • TodoWrite 清空
        • 不产生任何持久化文件
```

## 数据模型

### TodoWrite 任务结构

```typescript
interface NewfeatureTask {
  activeForm: string;    // "正在实现数据层"
  content: string;       // "数据层：添加搜索参数"
  status: 'pending' | 'in_progress' | 'completed';
}
```

### Git 提交格式

```
<type>(newfeature): <任务描述> - <任务块名称>

<改动详情>

Co-Authored-By: OpenMatrix
```

## 验证策略

### 按质量等级执行

| 质量等级 | 测试 | Lint | 覆盖率 | 安全扫描 | E2E |
|---------|:---:|:----:|:------:|:--------:|:---:|
| strict | ✅ | ✅ | >80% | ✅ | 用户选择 |
| balanced | ✅ | ✅ | >60% | ✅ | 用户选择 |
| fast | ❌ | ❌ | 无 | ❌ | ❌ |

### 验证时机

- 每个小任务完成后立即执行验证（除 fast 模式）
- 全部完成后执行整体验证

## 错误处理

| 错误场景 | 处理方式 |
|---------|---------|
| 测试失败 | 停止执行，提示用户修复，不自动重试 |
| Lint 错误 | 停止执行，提示用户修复 |
| Agent 执行超时 | 提示用户，询问是否重试或跳过 |
| Git 提交失败 | 提示用户手动处理 |

## 与现有指令的关系

| 指令 | 适用场景 | 任务文件 | 问答 |
|-----|---------|:-------:|:----:|
| `/om:newfeature` | 小需求（≤100字，单一功能） | ❌ | 质量+E2E |
| `/om:start` | 标准任务 | ✅ | 质量+E2E+模式 |
| `/om:brainstorm` | 复杂任务（架构/选型） | ✅ | 设计问答 |
| `/om:auto` | 全自动执行 | ✅ | 无问答 |

## 验收标准

1. `/om:newfeature` skill 文件创建完成
2. `/om` 主入口路由逻辑增加 AI 复杂度判断
3. 小需求正确路由到 `/om:newfeature`
4. 任务拆分为 2-5 个小块，用 TodoWrite 管理
5. 按质量等级执行验证
6. 每个小任务完成后分步 Git 提交
7. 不产生 `.openmatrix/tasks/` 文件
8. E2E 测试选项正确传递给验证流程

## 示例

**输入：**
```
/om 给用户列表页添加搜索功能
```

**路由判断：**
- 描述简短（≤100字）
- 单一功能点
- 不涉及架构
- → 路由到 `/om:newfeature`

**拆分：**
```
1. 数据层：添加搜索参数到 UserQuery 接口
2. API 层：实现搜索 API 端点
3. UI 层：添加搜索框组件和交互
```

**问答：**
- 质量等级：balanced
- E2E：不需要

**执行：**
```
Task 1: Agent 执行 → 验证 → Git 提交
Task 2: Agent 执行 → 验证 → Git 提交
Task 3: Agent 执行 → 验证 → Git 提交 → 总结
```
