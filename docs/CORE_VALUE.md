# OpenMatrix 核心价值与底层逻辑

## 项目定位

OpenMatrix 不做 AI，不做 IDE，定位为 **AI 编码的流程编排层**。

在 AI 编码工具的发展路径中：

```
第一层：AI 补全代码（Copilot）→ 解决「写」的问题
第二层：AI 对话编程（Claude Code）→ 解决「复杂任务」的问题
第三层：流程编排 + 质量保障（OpenMatrix）→ 解决「可信交付」的问题
```

OpenMatrix 处于第三层。不是让 AI 写得更好，而是让 AI 的产出 **可信、可交付**。

## 核心价值

**把软件工程的流程纪律，注入到 AI 编码的过程中。**

这不是技术问题，是信任问题。用户不放心让 AI 全自动，因为没有流程保障。质量门禁、多阶段验证、Accept review，本质都在解决可信度。Meeting 机制解决的也是信任：不确定的决策不擅自做，留给用户。

> 用流程纪律把 AI 的不确定性，转化为用户可信任的确定性产出。

## 底层逻辑闭环：三个嵌套飞轮

### 飞轮 1：质量飞轮（单次执行内）

```
不确定性输入（用户需求）
      │
      ▼
  拆解：大任务 → 小任务（降低单点风险）
      │
      ▼
  流水线：TDD → Develop → Verify → Accept（层层过滤）
      │
      ▼
  质量门禁：7 道检查（编译/测试/覆盖率/Lint/安全/E2E/验收）
      │
      ▼
  确定性输出：通过验证的代码 + 质量报告
```

原理：风险分散 + 层层过滤。单次 AI 调用的成功率不是 100%，但经过流水线过滤后，输出的可信度大幅提升。

### 飞轮 2：信任飞轮（跨次使用）

```
用户看到质量报告（量化、透明）
      │
      ▼
  信任提升 → 愿意委托更复杂的任务
      │
      ▼
  系统处理更复杂的场景 → 积累更多模式
      │
      ▼
  任务拆解更精准 → 成功率更高 → 报告更好看
      │
      ▼
  信任进一步提升 → ...
```

原理：信任来源于可量化、可验证。质量报告让用户「看见」质量，而不是「感觉」质量。和 CI/CD 之所以被信任是同一个道理。

### 飞轮 3：效率飞轮（对比传统模式）

```
传统模式：AI 写代码 → 用户手动检查 → 用户手动测试 → 用户手动修 bug
                              ↑ 重复 N 次 ↑

OpenMatrix：AI 写代码 → 自动验证 → 自动测试 → 自动修 → 只有 Meeting 需要人
```

人工介入次数从 N 次降到 0~2 次（只有 Meeting 需要用户参与）。

## 能够成功的四个基础条件

### 1. 趋势对

AI 编码已经过了「能不能写」的阶段，进入「敢不敢用」的阶段。用户痛点从 AI 能力转向 AI 可靠性。

### 2. 切入点对

不造 AI、不造 IDE，做流程编排层。不和模型厂商竞争，不和 IDE 竞争，是模型和 IDE 之间的必要桥梁。

类比：CI/CD 不写代码、不部署服务器，但它是代码到生产之间的必要流程。OpenMatrix 是 AI 到可信产出之间的必要流程。

### 3. 架构对

委托模型 + 持久化状态：
- AI 模型升级，OpenMatrix 自动变强
- 不受上下文窗口限制（step/complete 持久化循环）
- 不绑定特定模型（架构可复用到其他 AI Agent）

### 4. 哲学对

auto 模式不追求 100% 完成，追求 **最大前进距离**：
- 承认 AI 有不确定性
- 不假装完美，而是用流程管理不确定性
- 用户得到「可预期的部分完成」，而不是「不可预期的全部失败」

## 核心设计决策

### 委托模型：不执行，只编排

OpenMatrix 自己不写代码、不跑测试、不读文件。它只做决策（下一步做什么、分配给谁、质量标准是什么）和协调（管理状态、处理阻塞、流转阶段）。所有实际执行委托给 Claude Code 原生 Agent 系统。

TypeScript 代码是骨架，Prompt 模板是肌肉。

### 持久化 step/complete 循环：对抗上下文压缩

```
Skill 调用 openmatrix step → 拿到任务描述 → Agent 执行 → Skill 调用 openmatrix complete → 标记完成
     ↑                                                                                    │
     └────────────── 下一次 step，从磁盘读状态，不依赖上下文 ─────────────────────────────────┘
```

执行循环不依赖对话记忆。上下文压缩、崩溃、重启都不影响。

### Meeting 机制：最大前进距离

遇到无法抉择的问题 → 跳过当前任务 → 记录到 Meeting → 继续跑其他任务 → 最后用户集中处理。

auto 模式完全无人值守，所有 Meeting skip。系统追求最大前进距离，接受部分完成。

### 编排粒度：任务级编排

每个任务 = 一次 subagent 调用。不做 subagent 内部检查点。如果需要检查点，说明任务太大，应该在 Planner 层拆成更小的原子任务。质量保障由 phase 系统负责。

## 总结

```
可靠性 = 委托执行（跟随模型升级）
       + 持久化循环（不受上下文限制）
       + 最大前进距离（阻塞不中断）

价值 = 自动化程度 × 质量保障 / 人工介入次数
```

OpenMatrix 的成功逻辑 = 在 AI 能力爆发但信任缺失的窗口期，用软件工程的流程纪律填补「AI 能写」到「用户敢用」之间的信任鸿沟。
