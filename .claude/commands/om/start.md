---
name: om:start
description: 启动新的任务执行周期
---

<NO-OTHER-SKILLS>
执行此技能时，不得调用 superpowers、gsd 或其他任务编排相关的技能。OpenMatrix 独立运行，不依赖外部任务编排系统。
</NO-OTHER-SKILLS>

<objective>
解析任务文档，通过 3-7 个问题澄清需求，确认后启动执行。
</objective>

<process>
1. **检查当前状态**
   - 如果已有运行中的任务，提示用户先完成或暂停

2. **解析任务输入**
   - 如果提供文件路径 → 读取文件内容
   - 如果提供任务描述 → 直接使用
   - 如果无参数 → 询问用户要执行的任务

3. **生成澄清问题 (3-7个)**

根据任务内容，生成以下类型的问题：

```
📋 任务分析完成，请回答以下问题：

[1] 🎯 核心目标
    这个任务的主要目标是什么？
    A) 实现新功能
    B) 修复 Bug
    C) 重构优化
    D) 其他: ___

[2] 📦 技术栈
    使用什么技术栈？
    A) TypeScript + React
    B) Python + FastAPI
    C) Go + Gin
    D) 其他: ___

[3] 🗄️ 数据存储
    需要什么类型的存储？
    A) PostgreSQL
    B) MongoDB
    C) SQLite (本地)
    D) 无需存储

[4] 🔐 认证方式
    需要用户认证吗？
    A) JWT Token
    B) OAuth 2.0
    C) Session Cookie
    D) 无需认证

[5] 📡 API 风格
    API 采用什么风格？
    A) RESTful
    B) GraphQL
    C) gRPC
    D) 混合

[6] 🧪 测试要求
    测试覆盖率要求？
    A) >80% (严格)
    B) >60% (标准)
    C) >40% (基础)
    D) 无要求

[7] 📝 文档要求
    需要什么级别的文档？
    A) 完整文档 (API + 使用指南 + 架构)
    B) 基础文档 (README + API)
    C) 最小文档 (README)
    D) 无需文档

请逐个回答 (例如: 1A 2A 3C 4D 5A 6B 7B): _
```

4. **任务拆解**
   - 根据用户选择拆解任务
   - 生成子任务列表和依赖图

5. **展示执行计划**

```
📋 执行计划

## Phase 1: 设计阶段
  └─ TASK-001: 架构设计 (15min)

## Phase 2: 开发阶段
  ├─ TASK-002: 数据模型 (20min)
  ├─ TASK-003: API 接口 (30min)
  └─ TASK-004: 前端页面 (40min)

## Phase 3: 测试阶段
  ├─ TASK-005: 单元测试 (20min)
  └─ TASK-006: 集成测试 (15min)

## Phase 4: 收尾阶段
  ├─ TASK-007: 文档编写 (15min)
  └─ TASK-008: 代码审查 (10min)

📊 统计
  总任务: 8
  预计耗时: ~2.5小时
  审批点: plan, merge
```

6. **执行模式确认**

```
⚙️ 执行模式选择

[1] 🔔 每阶段确认 (推荐)
    每个阶段完成后暂停，等待你确认
    适合重要任务，可控性高

[2] 🚀 关键节点确认
    仅在 plan/merge/deploy 时暂停
    平衡速度和控制

[3] ⚡ 全自动执行
    无需确认，自动完成所有任务
    适合简单、低风险任务

请选择执行模式 [1-3]: _
```

7. **开始执行**
   - 用户确认后，调用 CLI: `openmatrix start`
   - CLI 根据选择的模式配置审批点

</process>

<arguments>
$ARGUMENTS
</arguments>

<examples>
/om:start                      # 交互式输入任务
/om:start docs/task.md         # 从文件读取任务
/om:start "实现用户登录功能"    # 直接描述任务
</examples>

<notes>
## 问题生成规则

问题应该覆盖：
1. **目标澄清** - 明确要做什么
2. **技术选择** - 确定技术栈
3. **范围界定** - 包含/排除什么
4. **质量要求** - 测试、文档标准
5. **约束条件** - 时间、资源限制
6. **风险识别** - 潜在问题和依赖

## 执行模式对比

| 模式 | 审批点 | 适用场景 |
|------|--------|---------|
| 每阶段确认 | 每阶段结束 | 重要任务 |
| 关键节点确认 | plan/merge/deploy | 常规任务 |
| 全自动执行 | 无 | 简单任务 |
</notes>
